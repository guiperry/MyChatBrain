Clerk Implementation Guide

How to correctly integrate Clerk authentication into MyChatBrain, replacing the current custom JWT system. This document captures all the touch-points specific to this codebase.

Current State
Clerk was partially wired in then removed. The custom JWT system (src/lib/auth.ts, middleware.ts) is fully functional. Switching to Clerk requires replacing or bridging all five areas documented below.
1

Install & Environment Variables

Install the correct Clerk package for Next.js App Router. The project uses Next.js 14.

terminal
npm install @clerk/nextjs

Add these variables to your .env.local and to your Vercel project environment variables. Without them the build will succeed but ClerkProvider will throw at runtime.

.env.local
NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY=pk_live_xxxxxxxxxxxx
CLERK_SECRET_KEY=sk_live_xxxxxxxxxxxx

# Optional — redirect targets after sign-in / sign-out
NEXT_PUBLIC_CLERK_SIGN_IN_URL=/login
NEXT_PUBLIC_CLERK_SIGN_UP_URL=/register
NEXT_PUBLIC_CLERK_AFTER_SIGN_IN_URL=/
NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL=/
Where to get the keys
Go to clerk.com → Dashboard → your app → API Keys. The Publishable Key starts with pk_ and is safe to expose to the client. The Secret Key starts with sk_ and must stay server-side only.

Replace middleware.ts with Clerk Middleware

📄 middleware.ts (project root)

The current middleware.ts manually checks the gemini-auth-token JWT cookie and redirects unauthenticated users. Clerk's middleware does the same thing declaratively. Replace the entire file:

middleware.ts
import { clerkMiddleware, createRouteMatcher } from '@clerk/nextjs/server';

// Routes that are publicly accessible without authentication
const isPublicRoute = createRouteMatcher([
  '/login(.*)',
  '/register(.*)',
  '/api/auth/login',
  '/api/auth/register',
  '/api/health',
]);

export default clerkMiddleware(async (auth, request) => {
  if (!isPublicRoute(request)) {
    await auth.protect();
  }
});

export const config = {
  matcher: [
    // Skip Next.js internals and static files
    '/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)',
    '/(api|trpc)(.*)',
  ],
};
Remove the old custom middleware logic
Once Clerk middleware is active, src/lib/auth.ts's authMiddleware function and the gemini-auth-token cookie check are no longer needed at the middleware layer. The custom auth utilities (verifyToken, comparePasswords) can be retired or kept if you want a migration period.

Update API Routes to Use Clerk Auth

Every API route that currently calls verifyToken() or reads gemini-auth-token from cookies needs to switch to Clerk's server-side auth() helper. Here's the pattern:

Current pattern (custom JWT)

src/app/api/auth/me/route.ts — current
import { verifyToken } from '@/lib/auth';
import { cookies } from 'next/headers';

export async function GET(request: NextRequest) {
  const cookieStore = cookies();
  const token = cookieStore.get('gemini-auth-token')?.value;
  if (!token) return NextResponse.json({ error: 'Not authenticated' }, { status: 401 });

  const decoded = verifyToken(token);
  const user = await dbHelper.getUser(decoded.userId);
  // ...
}

Replacement pattern (Clerk)

src/app/api/auth/me/route.ts — with Clerk
import { auth } from '@clerk/nextjs/server';

export async function GET(request: NextRequest) {
  const { userId } = await auth();
  if (!userId) return NextResponse.json({ error: 'Not authenticated' }, { status: 401 });

  // userId is a Clerk string ID like "user_2abc..."
  // See Step 5 for bridging it to your NebulaDB numeric user ID
  const dbUser = await dbHelper.getUserByClerkId(userId);
  // ...
}

Routes that need updating

File Current auth check Action
api/auth/me/route.ts verifyToken from cookie Replace with auth()
api/auth/login/route.ts bcrypt compare + JWT sign Delete — Clerk handles login
api/auth/logout/route.ts Deletes cookie Delete — use Clerk's SignOutButton
api/auth/register/route.ts bcrypt hash + DB insert Delete — Clerk handles registration
api/createSession/route.ts verifyToken from cookie Replace with auth()
api/saveChat/route.ts verifyToken (if present) Replace with auth()
api/loadChats/route.ts verifyToken (if present) Replace with auth()
5

Bridge Clerk User IDs to NebulaDB

This is the most important architectural decision. Clerk identifies users with string IDs like user_2abc123xyz. Your NebulaDB tables use numeric user_id foreign keys. You have two options:

Option A — Add a clerk_id column to users (Recommended)

Add a clerk_id column to the users table. On first login after migration, look up the user by their Clerk ID. If not found, create a DB record and store the Clerk ID. All existing foreign key relationships stay intact.

src/db/nebuladb-helper.ts — add this method
async getUserByClerkId(clerkId: string): Promise<DbUser | null> {
  const users = await this.db.find('users', { clerk_id: clerkId });
  return users[0] ?? null;
}

async upsertClerkUser(clerkId: string, email: string, name: string): Promise<DbUser> {
  let user = await this.getUserByClerkId(clerkId);
  if (!user) {
    user = await this.createUser({ clerk_id: clerkId, email, username: name, password: '' });
  }
  return user;
}

Option B — Use Clerk's userId as the primary key

Change the DB schema to use clerk_id TEXT PRIMARY KEY instead of a numeric ID. More work upfront but eliminates the bridge permanently. Only practical if you are doing a fresh database setup with no existing data to migrate.

Clerk Webhook for sync
Set up a Clerk webhook at /api/webhooks/clerk that listens for user.created and user.deleted events. This keeps your NebulaDB in sync automatically without relying on "first login" upsert logic. Install svix (npm install svix) to verify webhook signatures.

Update ChatBody.tsx UI Components

📄 src/components/ChatBody.tsx

The header previously had SignInButton and UserButton from Clerk. Here is the correct way to use them, plus how to conditionally show content based on auth state using the useUser hook:

src/components/ChatBody.tsx — header section
"use client";
import { SignInButton, SignOutButton, UserButton, useUser } from '@clerk/nextjs';

// Inside the component:
const { isSignedIn, user, isLoaded } = useUser();

// In the JSX header:
<div className={styles.headerActions}>
  {!isLoaded && <span>Loading...</span>}

  {isLoaded && !isSignedIn && (
    <SignInButton mode="modal">
      <button className={styles.signInBtn}>Sign In</button>
    </SignInButton>
  )}

  {isLoaded && isSignedIn && (
    <UserButton afterSignOutUrl="/login" />
  )}
</div>
UserButton vs SignOutButton
UserButton renders Clerk's pre-built avatar dropdown (includes sign-out, profile management). SignOutButton is a plain wrapper you style yourself — better if you want a custom logout button in the sidebar.

Replace or Remove Login / Register Pages

The app has custom /login and /register pages. With Clerk you have two paths:

ApproachHowTrade-off
Clerk-hosted UI Delete the pages. Clerk redirects to its hosted sign-in flow at accounts.clerk.dev. Zero maintenance. Less control over appearance.
Embedded components Replace page content with <SignIn /> and <SignUp /> from @clerk/nextjs. Stays on your domain. Clerk handles all logic, you control the surrounding layout.
Custom UI + Clerk hooks Keep your existing forms, use useSignIn() and useSignUp() hooks to call Clerk's auth methods. Most control. Most work. Needed if you want a fully custom design.

Embedded component example (recommended)

src/app/login/page.tsx
import { SignIn } from '@clerk/nextjs';

export default function LoginPage() {
  return (
    <main style={{ display: 'flex', justifyContent: 'center', paddingTop: '80px' }}>
      <SignIn />
    </main>
  );
}
8

Clean Up Custom Auth Utilities

Once Clerk is fully wired in, the following files can be deleted or archived:

  • src/lib/auth.ts — hashPassword, comparePasswords, createToken, verifyToken, authMiddleware are all replaced by Clerk. The getCurrentUser function is replaced by auth() server-side or useUser() client-side.
  • src/app/api/auth/login/route.ts — Clerk handles credential verification.
  • src/app/api/auth/register/route.ts — Clerk handles user creation.
  • src/app/api/auth/logout/route.ts — Clerk handles session termination.
  • Dependencies to remove: bcryptjs, jsonwebtoken, @types/bcryptjs, @types/jsonwebtoken — no longer needed once JWT auth is gone.
