MyChatBrain — Embeddings Refactor Plan

Replace Pinecone + placeholder vectors with the custom landmark-lattice-v1 embedder at https://text-embedder-nine.vercel.app. Simultaneously migrate latent system-prompt context from raw text assembly to an indexed semantic retrieval pipeline.

Generated: 2026-06-01  ·  Branch: main  ·  Embedder model: landmark-lattice-v1 (768-dim)

0 — Motivation & Goals

Current state problems:

  • Fake embeddings. vectorstore.ts:getEmbedding() generates seeded random 1536-dim vectors. Similarity search returns semantically meaningless results.
  • Pinecone unused in practice. API keys are empty in all env files; the Pinecone integration has never run in production.
  • Raw-text system prompts. buildSystemPrompt() assembles the 15 most-recent nodes + top-5 interests without any semantic relevance to the user's actual message.
  • Token waste. Injecting all interests, goals, traits, and topics as raw sentences burns tokens on context that may be irrelevant to the turn.

After-refactor state:

  • All embeddings generated by https://text-embedder-nine.vercel.app (768-dim, landmark-lattice-v1).
  • Pinecone dependency completely removed. Embedding index lives in NebulaDB (embedding_index collection), persisted in Vercel Blob.
  • System prompt context is semantically retrieved: user message is embedded → top-K most relevant persona items from each category are fetched and injected.
  • Every new interest, goal, trait, and memory node extracted from a conversation is immediately indexed as an embedding.

1 — Custom Embedder API Reference

Base URL: https://text-embedder-nine.vercel.app  ·  All requests: Content-Type: application/json

EndpointMethodRequest BodyResponse Shape
/embed POST { "text": "..." } { embedding: number[768], dimensions: 768, model: string, token_count: number }
/embed/batch POST { "texts": ["...", "..."] } — max 128 { results: [{ index, embedding, dimensions, token_count }], model, count }
/similarity POST { "text_a": "...", "text_b": "..." } { similarity: number [0,1], model: string }
/health GET — Model metadata JSON
Embedding vectors are 768-element fixed-point int32 arrays. Each element is in [0, 10000]. Standard cosine-similarity applies directly — no normalization preprocessing is required.

2 — File-by-File Change Map

New src/lib/textEmbedder.ts

HTTP client for the custom embedder. Exports getEmbedding(text), getBatchEmbeddings(texts), and getTextSimilarity(a, b). This is the single source of truth for all embedding generation — no other file should call the embedder API directly.

New src/lib/embeddingIndex.ts

Local vector index backed by NebulaDB's embedding_index collection (persisted in Vercel Blob). Provides indexEntry(), deleteEntry(), searchIndex(), and reindexUserData(). Replaces all Pinecone read/write operations.

New src/app/api/memory/reindex/route.ts

POST endpoint that triggers full re-indexing of all persona data for the authenticated user. Useful after bulk imports, onboarding, or schema migrations.

Modified src/db/nebuladb.ts

Modified src/lib/memoryVectors.ts

3 — Step-by-Step Implementation Order

1

Create src/lib/textEmbedder.ts

Build the HTTP client first — everything else depends on it. Implement single embed, batch embed, and similarity endpoints. Hard-code the base URL as a constant with an env-var override (NEXT_PUBLIC_EMBEDDER_URL). Add a lightweight in-process LRU cache keyed on text so identical strings aren't re-embedded within the same request lifecycle.

2

Register embedding_index collection in nebuladb.ts

Add one line inside initCollections(): _collections.embedding_index = db.collection('embedding_index');. On first boot, NebulaDB creates the collection in Vercel Blob automatically.

3

Create src/lib/embeddingIndex.ts

Implement the local vector index layer over NebulaDB. The collection document shape is:

{
  id:         "interest-abc123",   // prefixed source id
  userId:     "42",
  category:   "interest",           // interest | goal | trait | memory_node | note
  text:       "machine learning",   // the original text that was embedded
  vector:     [4823, 7201, ...],     // 768-element int32 array
  metadata:   { weight: 0.87 },
  created_at: "2026-06-01T...",
  updated_at: "2026-06-01T..."
}

The searchIndex() function loads all documents for a user (filtered by optional category list), computes cosine similarity between the query vector and each stored vector, and returns top-K results above a configurable threshold.

NebulaDB does not support approximate nearest-neighbour indexes. All similarity lookups are exact linear scans. This is acceptable at the scale of individual user persona data (typically <500 entries per user). If a user accumulates thousands of entries, consider batching or caching the scan results.
4

Update src/lib/vectorstore.ts — replace getEmbedding

Delete the seeded-random implementation (lines 120–149) and replace with:

export { getEmbedding } from './textEmbedder';

All other exports (LocalVectorStore, cosineSimilarity, normalize, etc.) are kept. cosineSimilarity in particular is now used by embeddingIndex.ts for local similarity scoring.

5

Rewrite src/lib/memoryVectors.ts — remove Pinecone

Replace the Pinecone client with calls to embeddingIndex.ts. All four exported functions keep identical signatures so callers need no changes:

  • storeNodeVector(nodeId, label, type, userId, metadata) → indexEntry(`node-${nodeId}`, userId, 'memory_node', label, { type, ...metadata })
  • deleteNodeVector(nodeId) → deleteEntry(`node-${nodeId}`, userId)
  • findSimilarNodes(query, userId, limit, threshold) → searchIndex(query, userId, { categories: ['memory_node'], topK: limit, threshold })
  • findRelatedNodes(nodeId, userId, limit, threshold) → fetch the entry's stored text, re-embed, search excluding the source node
6

Rewrite buildSystemPrompt() in chat/route.ts — semantic retrieval

The new flow:

Embed the user's message via getEmbedding(userMessage)

Query each category in parallel: top-3 interests, top-2 goals, top-2 traits, top-5 memory nodes — all ranked by cosine similarity to the user message vector

Assemble compact prompt from retrieved items only — items below the similarity threshold are silently omitted rather than padded with irrelevant context

Fallback: if the embedding index has no entries for this user (first session, fresh install), fall back to the old raw-text aggregation path so the app keeps working before any indexing has occurred

The resulting system prompt is shorter, more contextually relevant, and semantically matched to each individual conversation turn.

7

Update memory/extract/route.ts — index after extraction

After persisting each extracted interest, goal, or memory node to NebulaDB, call indexEntry() immediately. This keeps the embedding index current without requiring a manual reindex after every conversation.

These indexEntry() calls run in the background extraction request (already fire-and-forget from the chat route), so they add no latency to the user-facing chat response.
8

Create src/app/api/memory/reindex/route.ts

A POST endpoint that calls reindexUserData(userId) to bulk-embed all existing persona data. Used for initial migration of existing users, or after bulk data imports. Protected by the same auth cookie as all other memory routes.

POST /api/memory/reindex
→ { indexed: number, errors: number }
9

Remove Pinecone — cleanup

  • Delete src/lib/pinecone.ts
  • Remove @pinecone-database/pinecone from package.json and run npm uninstall @pinecone-database/pinecone
  • Remove pineconeKey, pineconeEnvironment, pineconeIndex from getenv/route.ts setting keys
  • Update LeftSidebar.tsx: remove the three Pinecone fields, add optional Embedder URL override field
  • Remove src/types/pinecone.d.ts if it exists
10

Trigger initial reindex for existing users

On first deploy, call POST /api/memory/reindex for any existing authenticated user so their persona data is indexed. This can be done manually via the settings panel, or automatically on first chat request when the embedding index is empty (cold-start detection).

4 — New Data Flow

── Chat Request ──────────────────────────────────────────────────────────

User message
  │
  ├─ getEmbedding(message)          ← textEmbedder.ts → /embed
  │    └─ queryVector: number[768]
  │
  ├─ searchIndex(queryVector, userId, { categories: ['interest'], topK: 3 })
  ├─ searchIndex(queryVector, userId, { categories: ['goal'],     topK: 2 })
  ├─ searchIndex(queryVector, userId, { categories: ['trait'],    topK: 2 })
  └─ searchIndex(queryVector, userId, { categories: ['memory_node'], topK: 5 })
       └─ all run in parallel via Promise.all()
            └─ ranked by cosine similarity, threshold 0.3

  ↓  semantically relevant context items only
  buildSystemPrompt() → compact structured string

  → routeToProvider(provider, message, systemPrompt)
  → response to user

── Background Extraction (fire-and-forget) ───────────────────────────────

  POST /api/memory/extract
    ├─ LLM extracts: topics, interests, goals, sentiment
    ├─ Persist to NebulaDB collections (unchanged)
    └─ For each new item:
         indexEntry(id, userId, category, text, metadata)
           └─ getEmbedding(text) → textEmbedder.ts → /embed
           └─ upsert to embedding_index collection in NebulaDB

── Reindex (on-demand) ────────────────────────────────────────────────────

  POST /api/memory/reindex
    └─ reindexUserData(userId)
         ├─ load all interest_metrics, goal_metrics, personality_traits, memory_nodes
         └─ getBatchEmbeddings(texts)  ← /embed/batch (up to 128 at a time)
              └─ upsert all to embedding_index

6 — New System Prompt Format

Before (raw text, always the same items regardless of conversation topic):

You are a personal AI assistant with persistent memory about this user.

Known interests: machine learning, cooking, travel, finance, gaming
Active goals: Learn Python | Lose 10 lbs | Save for vacation
Personality: openness:high, conscientiousness:medium
Recent topics: neural networks, Python, gym, budgeting, Tokyo, sushi

Use this context to personalize your responses naturally...

After (semantically retrieved, matched to the user's current message):

You are a personal AI assistant with persistent memory about this user.

Relevant context for this conversation:
• [interest] machine learning (relevance: 0.94)
• [interest] Python programming (relevance: 0.91)
• [goal] Learn Python — active (relevance: 0.89)
• [memory] neural networks discussion (relevance: 0.87)
• [memory] PyTorch tutorial question (relevance: 0.82)

Use this context to personalize your responses naturally — don't narrate or
recite it, just let it inform your tone, examples, and recommendations.
Relevance scores are cosine similarities between the user's current message embedding and each stored item's embedding. Items scoring below the configured threshold (default: 0.3) are never included, no matter how many entries exist in the index.
