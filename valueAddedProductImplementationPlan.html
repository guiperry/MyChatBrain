Track 1 · New Feature

Creator Algorithm — Session Persistence

Currently the Creator Algorithm is a 3-phase React state machine with no persistence. Navigating away or refreshing the page loses all progress. This track adds a full save/resume/list cycle backed by NebulaDB and surfaced in the right sidebar.

Root Cause
All state lives in useState inside CreatorAlgorithmApp. There is no API route, no DB collection, and no sidebar integration for sessions. The fix requires adding a DB collection, two API routes, updating the component, and updating the sidebar.

2
Create API routes for Creator sessions
src/app/api/creator-sessions/route.ts & [id]/route.ts

Two files needed — one for list/create, one for update/delete by ID.

src/app/api/creator-sessions/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { getNebulaDBHelper } from '@/db/nebuladb-helper';
import { verifyToken } from '@/lib/auth';
import { cookies } from 'next/headers';
import { initializeDatabase } from '@/db/nebuladb';

// GET /api/creator-sessions — list all sessions for current user
export async function GET(request: NextRequest) {
  const token = cookies().get('gemini-auth-token')?.value;
  const decoded = token ? verifyToken(token) : null;
  await initializeDatabase();
  const db = await getNebulaDBHelper();
  const sessions = await db.getCreatorSessions(decoded?.userId ?? 0);
  return NextResponse.json({ sessions });
}

// POST /api/creator-sessions — create or update a session
export async function POST(request: NextRequest) {
  const token = cookies().get('gemini-auth-token')?.value;
  const decoded = token ? verifyToken(token) : null;
  const body = await request.json();
  await initializeDatabase();
  const db = await getNebulaDBHelper();

  const session = await db.upsertCreatorSession({
    ...body,
    user_id: decoded?.userId ?? 0,
    updatedAt: new Date().toISOString()
  });
  return NextResponse.json({ session }, { status: 201 });
}
src/app/api/creator-sessions/[id]/route.ts
// DELETE /api/creator-sessions/[id]
export async function DELETE(request: NextRequest, { params }: { params: { id: string } }) {
  await initializeDatabase();
  const db = await getNebulaDBHelper();
  await db.deleteCreatorSession(params.id);
  return NextResponse.json({ success: true });
}

DB Helper Methods to add

src/db/nebuladb-helper.ts — add these three methods
async getCreatorSessions(userId: number) {
  const cols = await getTypedCollections();
  return (await cols.creator_sessions.find({ user_id: userId.toString() }).toArray()) ?? [];
}

async upsertCreatorSession(session: any) {
  const cols = await getTypedCollections();
  if (session.id) {
    await cols.creator_sessions.update({ id: session.id }, { $set: session });
    return session;
  }
  const id = `cs-` + Date.now().toString();
  const doc = { ...session, id, createdAt: new Date().toISOString() };
  return cols.creator_sessions.insert(doc);
}

async deleteCreatorSession(id: string) {
  const cols = await getTypedCollections();
  return cols.creator_sessions.delete({ id });
}
3
Update CreatorAlgorithmApp — auto-save & resume
src/components/CreatorAlgorithm.tsx

Add two props and three state-driven side effects to the existing component. The component already manages phase, context, systems, selectedSystem, constraint, and aiInsights — all of these need to be saved.

  • Add sessionId and onSessionSaved props so the parent can pass in a session to resume and receive the saved session ID back.
  • On mount, if sessionId is provided, fetch GET /api/creator-sessions, find the matching session, and restore all state fields (phase, context, systems, etc.).
  • Add a useEffect that watches phase — whenever phase advances (1→2, 2→3), call POST /api/creator-sessions with the current full state. Store the returned session id locally so subsequent saves become updates.
  • Auto-generate the session title from the first 60 characters of context.problem so it shows meaningfully in the sidebar.
  • When the user clicks "Start Over", call DELETE /api/creator-sessions/[id] if a session ID exists, or keep it as a completed session (user's choice via a confirmation dialog).
src/components/CreatorAlgorithm.tsx — new props + auto-save effect
interface CreatorAlgorithmProps {
  onClose?: () => void;
  sessionId?: string;           // Pass to resume a saved session
  onSessionSaved?: (id: string, title: string) => void;
}

// Inside the component — add after existing useState declarations:
const [activeSessionId, setActiveSessionId] = useState<string | null>(sessionId ?? null);

// Load existing session on mount
useEffect(() => {
  if (!sessionId) return;
  fetch('/api/creator-sessions')
    .then(r => r.json())
    .then(({ sessions }) => {
      const s = sessions.find((s: any) => s.id === sessionId);
      if (!s) return;
      setPhase(s.phase);
      setContext(s.context);
      setSystems(s.systems);
      setSelectedSystem(s.selectedSystem);
      setConstraint(s.constraint);
      setAiInsights(s.aiInsights);
    });
}, [sessionId]);

// Auto-save whenever phase, systems, or selectedSystem changes
useEffect(() => {
  if (phase === 1 && !activeSessionId) return; // nothing to save yet
  const title = context.problem.slice(0, 60) || 'Creator Session';
  fetch('/api/creator-sessions', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: activeSessionId,
      title, phase, context, systems, selectedSystem, constraint, aiInsights
    })
  })
  .then(r => r.json())
  .then(({ session }) => {
    if (!activeSessionId) {
      setActiveSessionId(session.id);
      onSessionSaved?.(session.id, title);
    }
  });
}, [phase, systems, selectedSystem, aiInsights]);

Track 2 · Critical

Close the Memory Loop

The most impactful single change in this entire plan. The memory_nodes, interest_metrics, goal_metrics, and personality_traits collections already store rich user data. This track makes the AI actually use it.

A
Inject memory context into every chat prompt
src/app/api/chat/route.ts

Before calling Gemini, fetch the user's memory data and prepend a system prompt. The user's JWT cookie is already available in the route — use it to look up their profile.

src/app/api/chat/route.ts — buildSystemPrompt()
async function buildSystemPrompt(userId: string): Promise<string> {
  const db = await getNebulaDBHelper();

  const [nodes, interests, goals, traits] = await Promise.all([
    db.getRecentMemoryNodes(userId, 15),
    db.getTopInterests(userId, 5),
    db.getActiveGoals(userId),
    db.getPersonalityTraits(userId),
  ]);

  if (!nodes.length && !interests.length) return '';

  return `You are a personal AI assistant with persistent memory about this user.

Known interests: ${interests.map(i => i.topic).join(', ')}
Active goals: ${goals.map(g => g.description).join(' | ')}
Personality: ${traits.map(t => t.trait).join(', ')}
Recent topics: ${nodes.map(n => n.label).join(', ')}

Use this context to personalize your responses naturally — don't narrate or recite it,
just let it inform your tone, examples, and recommendations.`;
}

// In the POST handler, before calling Gemini:
const systemPrompt = await buildSystemPrompt(userId);
const fullPrompt = systemPrompt
  ? `${systemPrompt}\n\nUser: ${message}`
  : message;

Return JSON only, no markdown:
{
  "topics": ["string"],
  "entities": ["string"],

Implementation Note
Fire the extraction with fetch('/api/memory/extract', { ... }) after returning the chat response. Use waitUntil on Vercel Edge functions, or simply don't await the fetch call on a serverless function — Vercel keeps the function alive briefly after the response is sent.

Track 4 · High Priority

Notes → Chat Context & Document RAG

A
Notes context injection from chat input
src/components/ChatBody.tsx + src/app/api/chat/route.ts

When a user types in the chat input, detect a @note: or /note prefix. Show a dropdown of matching saved notes. On selection, inject the note content into the prompt context.

  • In ChatBody.tsx, add a @-mention detector in the textarea's onChange handler. When @ is typed, fetch GET /api/notes/load and show a filterable dropdown.
  • When a note is selected, replace the @query with a [[note-id]] reference in the input.
  • In getChatResponse(), parse out any [[note-id]] references before sending, fetch those notes, and include them as context in the request body.
  • In /api/chat/route.ts, prepend injected notes to the prompt: "The user is referencing this note: [content]".
B
Complete the Pinecone RAG pipeline
src/app/api/upload/route.ts + src/lib/utils.ts

@pinecone-database/pinecone is installed. src/lib/utils.ts has indexGitHubRepo, extractGitHubURL, and runRAGChain. The upload endpoint exists. Three steps to make it work:

  • Upload: When a file is uploaded via /api/upload, chunk it, embed each chunk with Gemini embeddings (text-embedding-004), and upsert to Pinecone with user_id metadata.
  • Retrieve: On every chat request, embed the user's message and query Pinecone for the top-5 similar chunks owned by this user. Add them to the system prompt as "Relevant documents:".
  • UI: Add a file upload button to the chat input toolbar. Show a list of uploaded documents in the sidebar.
Env Vars Needed
Add PINECONE_API_KEY and PINECONE_INDEX to Vercel environment variables. Create a Pinecone index with dimension 768 (Gemini text-embedding-004).
